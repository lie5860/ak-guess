# 备份与还原功能（引继码）规格设计说明书

## 1. 目标
在已有的 `ak-guess` 游戏中，为玩家提供将其本地游戏记录（如测试报告、干员统计中已解锁的干员与胜率等关键数据）打包生成单一「引继码」上传至服务端的能力。同时允许玩家在客户端输入有效的引继码，从服务端获取存档云数据，展示二次确认面板，比对数据并覆盖本地数据。界面需借鉴 Cygames 旗下手游的美术与交互风格（暗调半透明磨砂质感、金属与高光修饰的组件卡片与按钮交互）。

## 2. 交互与 UI (Cy 系风格)
1. **入口**：在目前的「干员统计/图鉴」界面的合适位置（比如右上角），增加一个包含具体图标的【数据引继】按钮。
2. **引继模态框 (Modal)**：
   - 采用深色磨砂（`backdrop-filter: blur`），带有暗金色/金属蓝色内外光泽边框。
   - 顶部包含两个切换 Tab：**「生成引继码」** 与 **「使用引继码导入」**。
   - **生成 Tab 页面**：说明引继码的有效期（可选，交由后端决定），点击“立即生成”后前端调接口获取，并在屏幕核心区域放大展示 16 位字符（如 `XXXX-YYYY-ZZZZ-WWWW`），并附带快捷的**「一键复制」**按钮。
   - **导入 Tab 页面**：一个输入框与“查询”按钮。
3. **二次数据对比确认面板**：
   - 在用户点击查询且查得有效备份数据后，原位置淡入展现二次确认页面。
   - **左右双列比对列表**：左方显示【当前本地数据】，右方以发光边框或高亮文字显示【将要覆盖的云端数据】。
   - 展示的关键参考数据点：
     - 各模式累计胜利次数之和（涵盖 Daily、Random、Paradox等）
     - 已解锁的干员总数
     - 存档的最后更新/打包时间
   - 底部布置一个明显的、带有风险警示配色的「确认覆盖」加固按钮，并在点击后弹出版局级别的“覆盖成功并刷新页面”提示。

## 3. 数据层级 (Payload)
在向服务端传输和从服务端获取的数据体 (`payload`) 设计如下：

```json
{
  "lang": "zh_CN",
  "timestamp": 1700000000000,
  "recordData": {
    "daily": { ... },
    "random": { ... },
    "paradox": { ... }
  }
}
```

依赖于当前 `/src/component/History.tsx` 中的 `loadRecordData(lang)` 所返回的核心持久化游戏数据集。内容涵盖全部 3 种子模式的 `playTimes`, `winTimes`, `roles` 等所有在本地储存的主干状态与图鉴数据。

## 4. 接口协议规约 (API Proposal & Mock)
建议后端提供以下两个极简 API。前端同时在本地会单独抽离出一个 `TransferService`。在真实接口对接通之前，我们先以在客户端用 `Promise` 模拟 Mock 异步返回引继码与查询的形式，保证完整的 UI 流畅度和确认逻辑跑通。

1. **生成接口 (Generate)**
   - `POST /api/transfer/generate`
   - Body: 组装好的 JSON `<payload>`。
   - Response: `{ "code": 0, "data": { "transferCode": "A1B2-C3D4-E5F6-G7H8" } }`
2. **查询接口 (Query & Fetch)**
   - `GET /api/transfer/query?code=A1B2-C3D4-E5F6-G7H8`
   - Response: `{ "code": 0, "data": { "payload": { ...之前存入的所有数据... } } }`
